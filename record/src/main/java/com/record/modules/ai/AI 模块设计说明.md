# AI 模块设计说明

## 1. 文档目标

这份文档用于说明 `com.record.modules.ai` 模块在当前项目中的定位，以及后续如何把它逐步建设成一个相对完整、可扩展、适合企业级项目演进的 AI 模块。

你现在的目标不是一次性把所有 AI 能力都写完，而是先把模块边界设计清楚，然后按阶段落地：

1. 先把基础聊天能力做好。
2. 再接入会话记忆、Prompt 管理、调用日志。
3. 再做知识库、向量检索、RAG。
4. 最后做 Function Calling、Agent 编排、监控治理、安全控制。

这个顺序更适合初学者，也更符合真实项目的演进方式。

## 4. 企业级 AI 模块应该包含什么

一个完整的企业级 AI 模块，通常不仅仅是“聊天接口”，而是下面这些能力的组合。

### 4.1 Chat 对话层

这是最基础的一层，负责：

- 普通聊天
- 流式输出
- 多轮对话
- 会话上下文管理
- 系统提示词注入
- 参数控制，比如温度、最大输出长度、模型切换

你现在已经有一部分了。

### 4.2 Prompt 管理层

Prompt 不应该全部写死在 Java 代码里，应该做成可维护资源。

推荐能力：

- 系统 Prompt 模板
- 场景 Prompt 模板
- 变量占位符替换
- Prompt 版本管理
- Prompt 灰度发布
- Prompt 审核与回滚

你现在已经有 `prompts/ai` 目录，这是一个正确方向。后面可以把“账单分析”“知识库问答”“Agent 助手”分别拆成独立 Prompt。

### 4.3 Memory 会话记忆层

记忆分成两类：

- 短期记忆：当前会话上下文，通常放 Redis
- 长期记忆：用户长期偏好、常用信息、历史摘要，通常放数据库或知识库

你现在实现的是短期记忆，已经可以满足基础聊天场景。后续可以增加：

- 会话摘要压缩
- 长对话自动裁剪
- 用户画像记忆
- 用户偏好记忆

### 4.4 RAG 检索增强生成层

RAG 是企业项目非常核心的一层，它解决的问题是：

- 大模型不知道你的私有数据
- 模型幻觉严重
- 需要基于你的业务知识回答问题

RAG 的标准流程：

1. 导入文档
2. 文档切片
3. 向量化
4. 存入向量库
5. 用户提问时先检索相关片段
6. 将检索结果拼到 Prompt
7. 再交给大模型生成答案

对于你的项目，RAG 可用于：

- 个人知识库问答
- 日记内容检索总结
- 账单规则说明
- 系统使用帮助
- 个人生活记录总结

### 4.5 Vector Store 向量存储层

RAG 背后必须有向量库。

常见选择：

- 开发阶段：`pgvector`、Redis Vector、Milvus Lite
- 正式项目：`pgvector`、Milvus、Elasticsearch 向量检索

对于你这个项目，我建议优先级如下：

1. 如果你想简单一点，优先 `pgvector`
2. 如果你后续文档量很大，再考虑 `Milvus`

原因：

- 你本身已经是 Java + MySQL/Redis 思路
- `pgvector` 对初学者更容易理解和维护
- 企业项目里也很常见，不是玩具方案

### 4.6 Embedding 向量化层

向量库不能直接存文本，必须先做 Embedding。

这里要设计：

- 统一 Embedding 接口
- 模型切换能力
- 批量向量化能力
- 重试和失败补偿
- 重建索引任务

建议把 Embedding 单独抽象为一个服务，不要直接散落在业务代码里。

### 4.7 Function Calling 工具调用层

Function Calling 适合让模型“调用你的系统能力”，而不是只会聊天。

比如在你的 lifeRecord 项目里，模型未来可以调用：

- 查询日记
- 查询记账记录
- 查询提醒事项
- 创建提醒
- 获取某月消费统计
- 查询某天打卡情况
- 查询知识库文档

你现在已经有一个 `list_diaries` Demo，这是很好的起点。

企业级设计重点：

- 工具定义标准化
- 参数校验
- 工具权限控制
- 工具执行超时
- 工具结果审计
- 工具失败降级

### 4.8 Agent 编排层

当工具变多、场景变复杂后，就不仅是 Function Calling，而是 Agent。

Agent 负责：

- 任务拆解
- 工具选择
- 多步执行
- 结果汇总
- 失败恢复

例如：

- “帮我总结最近 7 天的日记和消费情况，再给我一个生活建议”

这个需求可能会拆成：

1. 查日记
2. 查账单
3. 汇总
4. 生成建议

这就属于 Agent 编排能力。

### 4.9 Observability 可观测性

企业项目里 AI 模块必须可观测，否则线上出问题很难查。

至少要记录：

- 用户 ID
- 会话 ID
- 场景
- 模型名称
- 输入 Token
- 输出 Token
- 耗时
- 是否成功
- 错误信息
- 工具调用次数
- RAG 检索命中情况

你现在已经有 `AiCallLog`，后续可以继续增强字段。

### 4.10 Security 安全治理

AI 模块是高风险模块，不能只考虑功能。

必须考虑：

- Prompt 注入攻击
- 越权访问用户数据
- 恶意工具调用
- 敏感信息泄露
- 内容安全审核
- 调用频率限制
- 成本控制

对你的项目尤其重要的一点是：

- AI 调用业务工具时，必须始终带上当前登录用户 `userId`
- 绝对不能让模型自己决定访问哪个用户的数据

你现在在 `DiaryListTool` 里把 `userId` 固定在服务端，这个思路是对的，要继续保持。

## 5. 推荐的 AI 模块分层

建议你把 AI 模块逐步拆成下面几层：

```text
com.record.modules.ai
├─ controller          // 对外接口
├─ application         // 场景编排层
├─ service             // 领域服务
├─ client              // 模型客户端封装
├─ prompt              // Prompt 加载与渲染
├─ memory              // 会话记忆
├─ rag                 // 检索增强
├─ embedding           // 向量化
├─ vector              // 向量库访问
├─ tool                // Function Calling 工具
├─ agent               // Agent 编排
├─ mapper              // 持久层
├─ model               // DTO/Entity/VO
└─ config              // 配置类
```

### 5.1 每层职责建议

`controller`

- 接收请求
- 参数校验
- 返回统一响应

`application`

- 负责一个完整场景的流程编排
- 例如“知识库问答流程”“账单分析流程”“生活助手流程”

`service`

- 封装通用业务能力
- 比如聊天服务、日志服务、会话服务

`client`

- 封装 `ChatClient`、`EmbeddingModel`
- 屏蔽底层模型供应商差异

`prompt`

- 管理 Prompt 模板
- 渲染变量

`memory`

- 管理会话上下文
- 管理摘要与长期记忆

`rag`

- 文档解析
- 文档切片
- 检索召回
- 重排
- 引用组装

`tool`

- 定义可调用工具
- 每个工具单一职责

`agent`

- 多工具编排
- 多阶段决策

## 7. RAG 模块建议设计

如果你准备做得全面，RAG 建议单独分包。

推荐结构：

```text
com.record.modules.ai.rag
├─ loader       // 文档加载，如 pdf、md、txt、网页
├─ parser       // 解析
├─ splitter     // 文本切片
├─ embedding    // 向量化
├─ store        // 向量库存储
├─ retriever    // 检索召回
├─ reranker     // 重排
├─ assembler    // 上下文组装
└─ citation     // 引用生成
```

### 7.1 一个标准 RAG 请求流程

1. 用户提问
2. 对问题改写或扩展检索词
3. 召回 TopK 文档片段
4. 可选重排
5. 组装上下文
6. 注入 Prompt
7. 调用大模型生成答案
8. 返回答案和引用来源

### 7.2 你应该保留引用信息

企业项目做 RAG，不要只返回答案，还要返回引用。

推荐响应里包含：

- 文档 ID
- 文档标题
- 片段内容摘要
- 相似度分数
- 片段位置

你现在已经有 `AiCitationVO`，很适合继续往这个方向做。

## 8. Function Calling 设计建议

Function Calling 不要做成“一个大工具类”，而应该一工具一职责。

推荐结构：

```text
com.record.modules.ai.tool
├─ diary
│  └─ ListDiariesTool.java
├─ ledger
│  └─ QueryMonthlyBillsTool.java
├─ reminder
│  ├─ ListRemindersTool.java
│  └─ CreateReminderTool.java
└─ registry
   └─ AiToolRegistry.java
```

### 8.1 工具设计原则

- 一个工具只做一件事
- 参数必须明确
- 参数必须校验
- 必须带用户上下文
- 返回结果要结构化
- 工具执行失败要可追踪

### 8.2 不建议让模型直接做的事

- 不要让模型直接拼 SQL
- 不要让模型自己传用户 ID
- 不要让模型直接访问 Mapper
- 不要让模型直接调用高风险写操作

正确方式应该是：

- 模型调用受控工具
- 工具内部再调用业务 Service

## 9. 配置设计建议

建议把 AI 配置拆清楚，至少包括下面几类：

```yaml
app:
  ai:
    enabled: true
    provider: openai
    model: gpt-4o-mini
    system-prompt: prompts/ai/system.md
    chat:
      memory-enabled: true
      max-messages: 20
    rag:
      enabled: true
      top-k: 5
      score-threshold: 0.7
    tool:
      enabled: true
      timeout-seconds: 10
    audit:
      enabled: true
```

后续还可以增加：

- embedding 模型配置
- 向量库配置
- 重排模型配置
- 速率限制配置
- 成本预算配置

## 11. 推荐开发路线

这是最重要的一部分。你现在是初学者，不建议一口气做完。

### 第一阶段：把现有聊天能力做扎实

目标：

- 聊天接口稳定
- 会话管理稳定
- Prompt 管理规范
- 日志记录完整

建议任务：

- 把 `AiServiceImpl` 里的会话逻辑拆到 `ConversationService`
- 把 AI 调用日志拆到 `AiLogService`
- 把 Function Calling Demo 独立成工具类
- 补充异常处理和返回码

### 第二阶段：补齐 Function Calling

目标：

- 不只是 Demo，而是正式工具体系

建议任务：

- 新增 `tool` 包
- 把日记、账单、提醒相关工具逐步接入
- 做工具注册中心
- 做工具权限控制

### 第三阶段：落地知识库与 RAG

目标：

- 支持上传文档
- 支持向量检索
- 支持带引用回答

建议任务：

- 新增知识库、文档、切片、检索相关表
- 先支持 Markdown / TXT 文档
- 先做简单切片 + TopK 检索

### 第四阶段：做 Agent 编排

目标：

- 一个请求调用多个工具

建议任务：

### 第五阶段：企业治理能力

## 13. 你现在最应该避免的问题

初学者做 AI 模块时，最容易踩下面这些坑：

### 13.1 所有逻辑都写进一个 ServiceImpl

这样后面一定会失控。

### 13.2 一开始就追求全功能

结果是每个功能都只有半成品。

### 13.3 Prompt 全写死在代码里

后期维护非常痛苦。

### 13.4 RAG 没有引用信息

用户会不信任答案，自己也难排错。

### 13.5 工具调用没有权限边界

这是严重安全问题。

### 13.6 会话只存在 Redis 不落库

短期可以，长期不够。

## 15. 下一步建议

如果你要继续做，我建议下一步直接开始下面两件事：

1. 先把 `AiServiceImpl` 拆分成聊天、会话、日志、工具四个部分。
2. 再正式新增 `rag` 和 `tool` 包结构，为后面扩展留好位置。

这样做的好处是：

- 你不会一下子失控

Milvus 接入改造步骤清单

一、先说当前项目现状

你现在的后端还没有真正接入向量库，知识库问答只是从 `knowledge_document` 表里取文档元数据和本地可读文本片段，再拼到 prompt 里给大模型。
所以它现在不是标准的 RAG，也没有 Embedding、向量入库、向量检索、TopK 召回、重排这些能力。

当前主要相关位置：
1. `D:\1\life-record\record\pom.xml`
   现在只有 `spring-ai-starter-model-openai`，还没有 Milvus 依赖。
2. `D:\1\life-record\record\src\main\resources\application.yaml`
   现在只有 OpenAI Chat 配置，Embedding 配置还是注释状态，也没有 Milvus 配置。
3. `D:\1\life-record\record\src\main\java\com\record\modules\knowledge\service\impl\KnowledgeServiceImpl.java`
   现在创建文档时只写了 `knowledge_document` 和 `knowledge_chunk_task`，没有真正做切片、Embedding、向量入库。
4. `D:\1\life-record\record\src\main\java\com\record\modules\ai\service\impl\AiServiceImpl.java`
   现在 `buildKnowledgeContext(...)` 还是直接查知识库文档并拼文本，没有走向量检索。


二、Milvus 接入后，系统应该变成什么样

标准流程应该是两条链路：

1. 写入链路
   上传文档 -> 解析正文 -> 文本切片 -> 调用 Embedding 模型生成向量 -> 写入 Milvus -> 回写文档状态

2. 查询链路
   用户提问 -> 生成问题向量 -> 去 Milvus 相似度检索 -> 拿到 TopK 片段 -> 拼到 prompt -> 大模型回答 -> 返回引用来源

你可以把它理解成：
MySQL 管“知识库、文档、任务、元数据”，Milvus 管“chunk 向量索引和召回”。


三、这个项目里要改哪里

第一处：Maven 依赖
文件：`D:\1\life-record\record\pom.xml`

建议增加：
1. `spring-ai-starter-vector-store-milvus`
   这是 Spring AI 官方的 Milvus VectorStore starter。
2. OpenAI Embedding 配置
   你已经用了 `spring-ai-starter-model-openai`，通常可以直接提供 `EmbeddingModel`，不用再单独换一套 SDK。

你最终至少要让项目具备两类 Bean：
1. `EmbeddingModel`
2. `VectorStore` 或 `MilvusVectorStore`


第二处：应用配置
文件：`D:\1\life-record\record\src\main\resources\application.yaml`

建议新增两组配置：

1. OpenAI Embedding 配置
建议放在：
`spring.ai.openai.embedding.options`

建议字段：
- `model`
- `dimensions`

2. Milvus 配置
如果走 Spring AI 官方 starter，建议放在：
`spring.ai.vectorstore.milvus`

常见字段：
- `client.host`
- `client.port`
- `client.username`
- `client.password`
- `databaseName`
- `collectionName`
- `embeddingDimension`
- `indexType`
- `metricType`
- `initialize-schema`

另外我建议你再单独加一组业务配置：
`app.ai.rag`

建议字段：
- `top-k`
- `similarity-threshold`
- `chunk-size`
- `chunk-overlap`
- `batch-size`
- `enabled`

这样“向量库连接配置”和“你自己业务的 RAG 策略”是分开的，后面更好维护。


第三处：新增 Milvus 配置类
建议目录：
`D:\1\life-record\record\src\main\java\com\record\modules\ai\config`

建议新增：
1. `AiRagProperties.java`
   绑定 `app.ai.rag.*`
2. `MilvusConfiguration.java`
   如果自动装配不够用，这里手动暴露 `VectorStore`、`MilvusServiceClient` 之类的 Bean

如果你先走 Spring AI 自动装配，这里甚至可以先不写复杂配置类，只保留一个 `AiRagProperties`。


第四处：补知识分片表
你当前只有：
- `knowledge_base`
- `knowledge_document`
- `knowledge_chunk_task`

建议再加一张：`knowledge_chunk`

建议至少包含字段：
- `id`
- `knowledge_base_id`
- `document_id`
- `user_id`
- `chunk_index`
- `content`
- `token_count`
- `vector_id`
- `status`
- `created_at`
- `updated_at`

原因很直接：
Milvus 里存的是向量检索索引，但你自己的业务还需要有一张关系表，能把“召回的向量”映射回“哪篇文档的哪一段内容”。

推荐做法：
- MySQL 保存 chunk 正文和元数据
- Milvus 保存 `vector + chunkId + knowledgeBaseId + documentId + userId` 这些可检索字段


第五处：改造知识库入库流程
核心文件：
`D:\1\life-record\record\src\main\java\com\record\modules\knowledge\service\impl\KnowledgeServiceImpl.java`

你现在创建文档时只是插入一条 PARSE 任务。真正接 Milvus 后，建议拆成下面几步：

1. 文档创建时只落库，不直接同步做重活
2. 后台任务消费 `knowledge_chunk_task`
3. 读取文档正文
4. 做清洗和切片
5. 每个 chunk 写入 `knowledge_chunk`
6. 调用 `EmbeddingModel.embed(...)` 生成向量
7. 批量写入 Milvus collection
8. 把返回的主键或业务主键回写到 `knowledge_chunk.vector_id`
9. 更新 `knowledge_document.status=VECTORIZED`
10. 任务状态改成 `SUCCESS`

这里最关键的点有两个：
1. 切片不要写死在 Controller 里，要封装成独立服务
2. 向量写入一定要批量，不要一个 chunk 调一次远程接口和一次向量库写入

建议新增服务：
- `KnowledgeChunkingService`
- `KnowledgeEmbeddingService`
- `KnowledgeVectorStoreService`
- `KnowledgeIngestionService`


第六处：改造知识库检索流程
核心文件：
`D:\1\life-record\record\src\main\java\com\record\modules\ai\service\impl\AiServiceImpl.java`

你现在最需要替换的是：
`buildKnowledgeContext(Long userId, Long knowledgeBaseId, String message)`

改造目标：
1. 用 `EmbeddingModel` 给用户问题生成 query vector
2. 去 Milvus 做 similarity search
3. 限定检索范围到当前知识库，必要时限定到当前用户
4. 根据召回结果拿到 chunkId 列表
5. 去 MySQL 查 `knowledge_chunk` 正文和文档信息
6. 组装 citations
7. 把 topK chunk 拼成最终上下文

这里建议你不要再自己手写“文件名匹配 + 文本 preview”那套逻辑了，直接替换成标准检索链路。


第七处：返回值也要升级
你现在前端已经有 `sources/citations` 面板了，所以后端最好把引用信息补完整。

建议 citations 至少返回：
- `chunkId`
- `documentId`
- `documentTitle`
- `snippet`
- `score`
- `knowledgeBaseId`

这样前端才能把“引用片段”和“命中来源”展示完整。


四、推荐的最小落地方案

如果你是第一次接 Milvus，我建议不要一上来就搞分布式和复杂工作流，先做一个最小可跑版本：

第 1 步：先把 Milvus 跑起来
开发环境建议：
- 本地 Docker 跑 Milvus Standalone

原因：
- 比 Distributed 简单很多
- 比 Lite 更接近你后面真实 Java 服务接入方式
- 对你现在这个 Spring Boot 项目最顺手

第 2 步：先打通 Embedding + 写入 + 查询
顺序建议：
1. 手动写死一段测试文本
2. 生成 embedding
3. 写入 Milvus
4. 用一条测试 query 做相似检索
5. 确认能召回正确 chunk

第 3 步：再接入知识库文档流
顺序建议：
1. 文档入库
2. 文本切片
3. 向量化
4. Milvus 入库
5. 问答检索
6. 引用回显

第 4 步：最后再做异步任务和重建索引
也就是把同步流程改成真正可重试、可重建、可删除向量的任务流。


五、建议你直接这样配置

`application.yaml` 可以按这个思路扩展：

1. OpenAI Embedding
- `spring.ai.openai.embedding.options.model=text-embedding-3-small` 或你实际使用的模型
- `spring.ai.openai.embedding.options.dimensions=1536`

2. Milvus
- `spring.ai.vectorstore.milvus.client.host=127.0.0.1`
- `spring.ai.vectorstore.milvus.client.port=19530`
- `spring.ai.vectorstore.milvus.client.username=root`
- `spring.ai.vectorstore.milvus.client.password=milvus`
- `spring.ai.vectorstore.milvus.databaseName=default`
- `spring.ai.vectorstore.milvus.collectionName=knowledge_chunk_vector`
- `spring.ai.vectorstore.milvus.embeddingDimension=1536`
- `spring.ai.vectorstore.milvus.indexType=IVF_FLAT`
- `spring.ai.vectorstore.milvus.metricType=COSINE`
- `spring.ai.vectorstore.milvus.initialize-schema=true`

3. 业务侧 RAG
- `app.ai.rag.enabled=true`
- `app.ai.rag.top-k=5`
- `app.ai.rag.similarity-threshold=0.65`
- `app.ai.rag.chunk-size=800`
- `app.ai.rag.chunk-overlap=120`
- `app.ai.rag.batch-size=20`

注意：
Embedding 维度必须和 Milvus collection 的向量维度一致，不一致会直接出问题。

八、几个最容易踩坑的点

1. Embedding 维度不一致
   这是最常见问题。模型维度和 Milvus collection 维度必须一致。
2. 切片太大
   chunk 太大，召回会变钝；太小，语义不完整。先用 500 到 1000 字符量级试。
3. 没有 metadata 过滤
   如果不按 `knowledgeBaseId` 或 `userId` 过滤，很容易串库。
4. 入库同步阻塞
   文档一上传就同步切片和 embedding，会拖垮接口响应。建议异步任务化。
5. 只存向量不存 chunk 正文
   后面引用来源、排错、重建索引都会很难做。
6. 直接在现有 `AiServiceImpl` 里堆所有逻辑
   后面会很难维护。切片、向量入库、检索要拆服务。

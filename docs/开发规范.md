# 开发规范

## 1. 文件编码

项目统一使用：

- `UTF-8`
- `无 BOM`
- 文本换行统一为 `LF`

适用范围：

- 后端 `Java / Yaml / SQL / Markdown`
- 前端 `Vue / TS / JS / JSON / Markdown`

原因：

- 后端 Java 文件如果带 `BOM`，Maven 编译时容易出现 `非法字符: ''\ufeff''`
- 中文注释在错误编码链路下容易变成乱码
- 统一 `LF` 可以减少跨平台无意义 diff

## 2. 注释规范

要求：

- 类、核心方法、关键字段要写注释
- DTO / VO / Entity 要补 `@Schema`
- 复杂逻辑前加简短单行注释
- 不写低价值注释，例如“给变量赋值”

建议：

- 能稳定使用中文时，优先中文注释
- 如果当前终端或脚本写入中文可能造成乱码，优先使用英文注释，保证源码稳定可读

## 3. 后端目录规范

后端模块统一按下面结构组织：

```text
modules/<module-name>/
  controller/
  service/
  service/impl/
  mapper/
  model/
    dto/
    vo/
    entity/
```

说明：

- `dto`：入参
- `vo`：出参
- `entity`：数据库实体

## 4. 编码写入规范

在当前仓库中，直接用某些终端命令写中文内容，可能造成二次转码乱码。

因此建议：

- 优先通过 IDE 保存文件
- 优先使用 `apply_patch` 修改源码
- 如果批量脚本写文件，必须确认输出为 `UTF-8 without BOM`

不要做的事：

- 不要随意用会写入 BOM 的脚本覆盖 Java 文件
- 不要在不确认编码的情况下批量替换中文注释

## 5. IDE 建议设置

IDEA / WebStorm / HBuilderX 建议统一设置为：

- File Encoding: `UTF-8`
- Project Encoding: `UTF-8`
- Default encoding for properties files: `UTF-8`
- 关闭 `native-to-ascii` 自动转换

如果发现乱码：

1. 先确认文件实际编码是否为 `UTF-8`
2. 再确认是否带 `BOM`
3. 不要直接继续编辑乱码文本，先恢复编码后再改

## 6. 提交前检查

提交前至少执行：

后端：

```bash
cd record
./mvnw.cmd -q -DskipTests compile
```

发布前：

```bash
cd record
./mvnw.cmd -q clean package
```

前端：

```bash
cd miniapp
npm run type-check
npm run build:mp-weixin
```

## 7. 本项目特殊约定

- 后端源码允许中文注释，但必须保证编码稳定
- 若某文件在当前环境下反复出现乱码，可以改成英文注释，优先保证可读性和可编译性
- `schema.sql` 必须写 `COMMENT`
- OpenAPI / Knife4j 模型说明必须补齐 `@Schema`

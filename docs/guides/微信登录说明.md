# 微信登录说明

## 1. 整体流程

```text
小程序前端                后端 /auth/wx-login              微信服务器
─────────                ─────────────────                ─────────
1. wx.login() 拿 code
2. POST code 给后端  ──►
                          3. WechatAuthClient
                              code2Session    ──────────►
                                                          ◄──── openid + session_key
                          4. 生成 access/refresh token
                          5. 写 user_session + Redis session
                       ◄── 返回 access/refresh
6. 存 token 到 storage
```

## 2. code2Session 调用细节

请求参数：
- `appid` ─ 小程序 AppID
- `secret` ─ 小程序 AppSecret
- `js_code` ─ 前端传来的 code
- `grant_type` ─ `authorization_code`

返回：
- `openid`（用户在该小程序唯一标识）
- `session_key`
- `errcode` / `errmsg`（失败时）

## 3. 为什么响应要先按 String 读

微信接口有时响应头标成 `text/plain`，但响应体内容是 JSON。
直接让 `RestTemplate` 反序列化对象会报：
```
no suitable HttpMessageConverter found
```

当前 `WechatAuthClient` 的处理：
1. 先拿原始字符串
2. 再用 `ObjectMapper` 手动解析成 `WechatCode2SessionResponse`

## 4. 常见报错含义

| 报错 | 含义 |
|---|---|
| `未获取到 openid` | 微信没正确返回用户身份 |
| `接口返回内容无法解析` | 微信返回内容不是合法 JSON |
| `微信登录失败: xxx` | 微信接口业务错误（appid/secret/code 有问题） |

## 5. 职责边界

`WechatAuthClient` 只负责"向微信换身份"。
真正生成系统里的 access/refresh token 在 `AuthServiceImpl.issueToken` 里完成。

## 6. 失败时的恢复策略

详见 `request.ts` 的 401 拦截器（2026-05-08 实现的三层兜底）：

1. 第一层：用 refreshToken 换新 access
2. 第二层：refresh 失败 → 自动 `wx.login()` + `/auth/wx-login` 静默重登
3. 第三层：彻底失败 → 跳 `/pages/auth/login` + toast 提示

## 1. 现状分析

### 1.1 已实现

| 层面 | 实现细节 |
|---|---|
| 认证（Authentication） | JWT 双 token（access + refresh） + 微信小程序登录 + OAuth 第三方登录（GitHub/Google） |
| 会话管理 | Redis Hash 存储 `sid` + `refreshToken`，TTL = refreshTokenExpireDays |
| 角色体系 | `sys_role` / `sys_user_role` / `sys_role_menu` 三张 RBAC 表，初始角色：`admin`（管理员）、`editor`（编辑者） |
| JWT → SecurityContext | `JwtAuthenticationFilter` 实时查库获取角色，写入 `ROLE_admin` / `ROLE_editor` |
| URL 级别授权 | `SecurityConfig` 配置：`/system/**` → `hasRole("admin")`，其余 → `authenticated()` |
| 管理后台菜单 | 菜单表 `sys_menu` 带 `admin_only` 字段，前端按角色过滤可见菜单 |
| Admin 引导 | `AdminBootstrap` 启动时自动给配置中的 userId 挂 admin 角色 |

### 1.2 安全漏洞

| 漏洞 | 说明 | 风险级别 |
|---|---|---|
| **无数据归属校验** | 任何登录用户能通过 `GET /diaries/detail/{id}` 访问任意其他用户的日记 | 🔴 高 |
| **无方法级权限控制** | 0 个 `@PreAuthorize` / `@Secured` / `@RolesAllowed` 注解，所有业务接口仅区分登录/未登录 | 🟠 中 |
| **无操作粒度鉴权** | 管理员删除用户、角色分配等敏感操作无额外权限校验，只靠 URL 模式 | 🟠 中 |
| **editor 角色未使用** | 数据库定义了 editor 角色但无任何代码引用 | 🟡 低 |

### 1.3 当前认证流程

```
微信小程序                     后端                         Redis
   │                           │                            │
   │  POST /auth/wx-login      │                            │
   │  ──────────────────────>  │                            │
   │                           │  wx.jscode2session ──>     │
   │                           │  <── openid, session_key   │
   │                           │                            │
   │                           │  生成 access+refresh JWT   │
   │                           │  存入 Redis (sid+refresh)  │
   │  <── {accessToken, refreshToken}                       │
   │                           │                            │
   │  GET /diaries/list        │                            │
   │  Authorization: Bearer <access>                        │
   │  ──────────────────────>  │                            │
   │                           │  JwtAuthenticationFilter:  │
   │                           │    1. 解析 JWT              │
   │                           │    2. 校验 sid ≠ Redis sid  │
   │                           │    3. 查 DB 获取角色        │
   │                           │    4. 写入 SecurityContext  │
   │                           │    5. 写入 UserContext      │
   │  <── 200 diary list       │                            │
```

## 2. 方案对比

### 2.1 可选方案

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **A. SecurityConfig URL 模式**（当前） | `requestMatchers(...).hasRole(...)` | 简单，集中管理 | 仅 URL 粒度；无法做数据归属；每次加接口都要改配置 | 只需区分登录/未登录、admin/普通 |
| **B. 方法级注解 `@PreAuthorize`** | 在 Controller 方法上加 SpEL 表达式 | 细粒度（方法级、数据归属）；声明靠接口代码就近维护；Spring Boot 标准做法 | 每个 API 要写注解；需写权限校验 Bean | 需要数据归属校验 + 角色控制（**本项目推荐**） |
| **C. 自定义 AOP 拦截器** | 自定义注解 `@RequiresOwner` + AOP 切面 | 完全自定义 | 重复造轮子；维护成本高 | Spring Security 方法安全不满足的极端场景 |
| **D. OAuth 2.0 权限框架** | spring-authorization-server 发第三方令牌 | 标准化授权码流程 | 过度设计，个人小程序无第三方接入需求 | 开放平台、SaaS 多租户 |

### 2.2 概念澄清：Spring Security vs OAuth 2.0

| | Spring Security | OAuth 2.0 |
|---|---|---|
| **本质** | **授权框架** — 定义谁能访问什么 | **认证/委托授权协议** — 让第三方应用代表用户操作 |
| **你项目里的用法** | 框架已集成，负责过滤器链、URL 规则 | 用于 GitHub/Google 第三方登录（`/auth/*/authorize`） |
| **能不能替代** | —— | ❌ 不能替代 Spring Security 做接口权限 |
| **关系** | Spring Security OAuth 2.0 模块是**建立在 Spring Security 之上**的 | |

**结论**：OAuth 2.0 解决的是"谁颁发的令牌"，不是"令牌能访问什么资源"。接口权限归 Spring Security 管。

### 2.3 推荐方案

**方案 B：`@PreAuthorize` 方法级注解**

理由：

1. 你的核心需求是**数据归属校验**（用户只能操作自己的日记/记账/打卡）+ **角色控制**（管理员操作系统接口），`@PreAuthorize` 天然覆盖
2. 已有 RBAC 表 + JwtFilter 角色注入，只差在 Controller 层用起来
3. 不改 SecurityConfig 核心逻辑，增量式改造
4. Spring Boot 3.x 官方推荐方案，社区文档丰富

## 3. 架构设计

### 3.1 总体架构

```
┌──────────────────────────────────────────────────────────┐
│                     请求入口                               │
└────────────────────┬─────────────────────────────────────┘
                     │
          ┌──────────▼──────────┐
          │  JwtAuthentication   │  ── 1. 解析 JWT
          │  Filter              │  ── 2. 校验 Redis sid
          │                      │  ── 3. 查角色写入 SecurityContext
          └──────────┬──────────┘
                     │
          ┌──────────▼──────────┐
          │  SecurityConfig     │  ── URL 级别粗规则
          │  (permitAll /        │     /auth/** → public
          │   hasRole/admin)    │     /system/** → admin
          │                      │     其余 → authenticated
          └──────────┬──────────┘
                     │
          ┌──────────▼──────────┐
          │  @PreAuthorize      │  ── 方法级细规则（新增）
          │  + SecurityExpression│     hasRole('admin')
          │  Root               │     @resourceAuthz.isOwner(#id)
          └──────────┬──────────┘
                     │
          ┌──────────▼──────────┐
          │  业务 Service       │  ── 正常业务逻辑
          └─────────────────────┘
```

三层鉴权各司其职，不重复、不冲突：

| 层 | 职责 | 例子 |
|---|---|---|
| **Filter 层**（JWT） | 认证：你是谁？ | 解析 token，写入 userId + roles |
| **URL 层**（SecurityConfig） | 粗粒度：这个路径要不要登录？要不要 admin？ | `/system/**` → admin only |
| **方法层**（@PreAuthorize） | 细粒度：这个资源你能不能操作？ | `@PreAuthorize("@diaryAuthz.isOwner(#id)")` |

### 3.2 数据归属校验流程

```
用户 A 请求 GET /diaries/detail/42
       │
       ▼
JwtFilter 写入 SecurityContext(userId=A, roles=[user])
       │
       ▼
@PreAuthorize("@diaryAuthz.isOwner(#id, authentication)")
       │
       ▼
diaryAuthz.isOwner(42, Authentication{principal=A}):
  1. 查 diary 表：diary.id=42 的 user_id
  2. 比较 user_id == A
  3. admin 角色直接放行（管理员可查看所有）
       │
   ┌───┴───┐
   │ true  │ false
   ▼       ▼
 放行     403 Forbidden
```

## 4. 数据库设计

### 4.1 现有 RBAC 表（无需改动）

```sql
-- 已有，不需要修改
sys_role          -- 角色表（admin, editor, ...）
sys_user_role     -- 用户-角色关联
sys_role_menu     -- 角色-菜单关联（管理后台动态路由）
sys_menu           -- 菜单表（含 admin_only 标志）
```

### 4.2 现有角色清单

| 角色名 | 说明 | 权限范围 |
|---|---|---|
| `admin` | 管理员 | 所有业务数据 + 系统管理接口 |
| `editor` | 编辑者 | 业务数据管理，不可操作系统配置（**暂未生效**） |

### 4.3 未来扩展：如需更细粒度的权限

当前 `sys_menu` 表的 `menu_type = 'BUTTON'` 行 + `auths` 字段已预留了按钮级权限标识位。如需细粒度控制（如"日记编辑按钮只有 editor 以上可见"），可启用：

```sql
INSERT INTO sys_menu (id, parent_id, name, path, menu_type, auths, title)
VALUES (1101, 11, 'DiaryAdd', '', 'BUTTON', 'diary:add', '新增日记');

INSERT INTO sys_menu (id, parent_id, name, path, menu_type, auths, title)
VALUES (1102, 11, 'DiaryDelete', '', 'BUTTON', 'diary:delete', '删除日记');
```

后端通过 `@PreAuthorize("hasAuthority('diary:add')")` 校验，前端根据 `auths` 字段控制按钮显示。

**但现阶段不需要急于做按钮级权限。数据归属校验优先级远高于此。**

### 5.2 Authz 组件接口设计

#### ResourceAuthz（通用基类）

```java
package com.record.security.authz;

/**
 * 资源归属校验基类。
 * 提供管理员跳过检查的通用逻辑，各模块 Authz 继承此类。
 */
public abstract class ResourceAuthz {

    /**
     * 判断当前用户是否是资源拥有者。
     *
     * @param resourceId 资源 ID
     * @param userId     当前登录用户 ID
     * @return true = 是拥有者或管理员
     */
    public abstract boolean isOwner(Long resourceId, Long userId);

    /**
     * 判断当前用户是否是资源拥有者，管理员也放行。
     * 在 SpEL 中以 @xxxAuthz.isOwner(#id, authentication) 形式调用，
     * 由 SecurityExpressionRoot 传入 authentication 中的 userId。
     *
     * @param resourceId 资源 ID
     * @param userId     当前登录用户 ID
     * @return true = 拥有者或管理员
     */
    public boolean isOwnerOrAdmin(Long resourceId, Long userId) {
        if (isAdmin(userId)) {
            return true;
        }
        return isOwner(resourceId, userId);
    }

    protected boolean isAdmin(Long userId) {
        // 由子类注入 RoleService 或通过 SecurityContext 判断
        // 推荐方案：直接从当前 SecurityContext 的 authorities 判断
        return SecurityContextHolder.getContext().getAuthentication().getAuthorities()
                .stream()
                .anyMatch(a -> a.getAuthority().equals("ROLE_admin"));
    }
}
```

#### DiaryAuthz（示例实现）

```java
package com.record.modules.diary.authz;

import com.record.modules.diary.mapper.DiaryMapper;
import com.record.modules.diary.model.entity.Diary;
import com.record.security.authz.ResourceAuthz;
import org.springframework.stereotype.Component;

/**
 * 日记归属校验组件。
 * 在 @PreAuthorize SpEL 表达式中通过 @diaryAuthz 引用。
 */
@Component("diaryAuthz")
public class DiaryAuthz extends ResourceAuthz {

    private final DiaryMapper diaryMapper;

    public DiaryAuthz(DiaryMapper diaryMapper) {
        this.diaryMapper = diaryMapper;
    }

    @Override
    public boolean isOwner(Long diaryId, Long userId) {
        Diary diary = diaryMapper.selectById(diaryId);
        return diary != null && diary.getUserId().equals(userId);
    }
}
```

> 其他模块 Authz 同理：`LedgerAuthz` 查 `ledger_entry.user_id`、`CheckinAuthz` 查 `checkin_task.user_id`、`MemorialAuthz` 查 `memorial.user_id`。

### 7.2 403 统一响应

在 `SecurityConfig` 中补充 `AccessDeniedHandler`：

```java
.exceptionHandling(exception -> exception
    .authenticationEntryPoint((request, response, authException) -> {
        // 已有：401 处理
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        // ...
    })
    .accessDeniedHandler((request, response, accessDeniedException) -> {
        // 新增：403 处理
        response.setStatus(HttpStatus.FORBIDDEN.value());
        response.setCharacterEncoding(StandardCharsets.UTF_8.name());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.getWriter().write(objectMapper.writeValueAsString(
                ApiResponse.failure(ErrorCode.FORBIDDEN.getCode(), "无权限访问")));
    }))
```

### 7.3 Controller 注解示例

```java
@RestController
@RequestMapping("/diaries")
@RequiredArgsConstructor
public class DiaryController {

    private final DiaryService diaryService;

    @PostMapping("/create")
    public ApiResponse<DiaryVO> create(@RequestBody @Valid DiaryCreateDTO dto) {
        // 创建时 userId 从 UserContext 获取，无需归属校验
        return ApiResponse.success(diaryService.create(dto, UserContext.getUserId()));
    }

    @PutMapping("/update/{id}")
    @PreAuthorize("@diaryAuthz.isOwner(#id, T(com.record.common.context.UserContext).getUserId())")
    public ApiResponse<DiaryVO> update(@PathVariable Long id, @RequestBody @Valid DiaryUpdateDTO dto) {
        return ApiResponse.success(diaryService.update(id, dto));
    }

    @GetMapping("/detail/{id}")
    @PreAuthorize("@diaryAuthz.isOwnerOrAdmin(#id, T(com.record.common.context.UserContext).getUserId())")
    public ApiResponse<DiaryVO> detail(@PathVariable Long id) {
        return ApiResponse.success(diaryService.detail(id));
    }

    @DeleteMapping("/delete/{id}")
    @PreAuthorize("@diaryAuthz.isOwner(#id, T(com.record.common.context.UserContext).getUserId())")
    public ApiResponse<Void> delete(@PathVariable Long id) {
        diaryService.delete(id);
        return ApiResponse.success(null);
    }
}
```

> **注意**：SpEL 中获取当前用户 ID 的方式。`@PreAuthorize` 执行时 `UserContext` 仍有效（在 Filter 的 `finally` 之前），可以直接用 `T(com.record.common.context.UserContext).getUserId()`。也可以通过 `authentication.principal` 传入（我们在 JwtFilter 中 principal 设的是 userId）：

```java
// 另一种写法，通过 authentication 传参
@PreAuthorize("@diaryAuthz.isOwner(#id, #root.authentication.principal)")
```

### 7.4 性能考量

`isOwner()` 每次校验会查库。优化方案：

1. **短期（推荐先做）**：直接查库，MyBatis-Plus 主键查询极快（<1ms）
2. **中期**：在 `ResourceAuthz` 层加 `@Cacheable`，Redis 缓存归属关系
3. **长期**：若并发量高，考虑在 JWT claims 中嵌入角色（避免每次查角色），同时归属关系走 Redis 缓存

现阶段直接查库即可，过度优化是浪费。

## 10. 术语表

| 术语 | 说明 |
|---|---|
| Authentication | 认证 — 验证"你是谁"（登录/ token 校验） |
| Authorization | 授权 — 验证"你能做什么"（角色/权限/数据归属） |
| RBAC | Role-Based Access Control — 基于角色的访问控制 |
| SpEL | Spring Expression Language — `@PreAuthorize` 中使用的表达式语言 |
| JWT | JSON Web Token — 无状态令牌，携带 userId/角色等 claims |
| SecurityContext | Spring Security 上下文，存储当前认证信息和角色 |
| `@PreAuthorize` | 方法调用前执行权限检查的注解 |
| `isOwner` | 自定义 SpEL 函数 — 校验资源归属当前用户 |
| `isOwnerOrAdmin` | 自定义 SpEL 函数 — 管理员跳过归属检查 |

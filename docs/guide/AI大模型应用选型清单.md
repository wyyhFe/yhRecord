# AI 大模型应用选型清单

## 1. 当前项目现状

当前项目技术基础：

- 后端：`Spring Boot + MyBatis-Plus + MySQL + Redis`
- 前端：`uni-app` 小程序
- AI 接入：`Spring AI`
- 已实现：
  - AI 聊天
  - AI 流式聊天
  - AI 账单分析
  - AI 账单分析历史

因此后续扩展“大模型应用”时，重点不是推倒重来，而是在现有架构上补齐：

- Prompt 管理
- 知识库
- 向量检索
- 工具调用
- 日志与监控

---

## 2. 选型总原则

选型建议按下面几层拆开看，不要混在一起：

1. **应用编排层**
   - 推荐：`Spring AI`

2. **业务主库**
   - 推荐：`MySQL`

3. **Prompt 模板管理**
   - 推荐：`MySQL`

4. **缓存 / 会话记忆**
   - 推荐：`Redis`

5. **知识库检索 / 向量库**
   - 这是重点选型项

---

## 3. 各层推荐结论

### 3.1 应用编排层

推荐：

- `Spring AI`

原因：

- 已经接入
- 和当前 Spring Boot 技术栈一致
- 支持：
  - ChatClient
  - Structured Output
  - Tool Calling
  - RAG
  - MCP

结论：

- **继续用 Spring AI，不建议更换框架**

---

### 3.2 业务主库

推荐：

- `MySQL`

存储内容：

- 用户
- 账单
- 日记
- AI 分析历史
- AI 调用日志
- Prompt 模板

原因：

- 当前项目已经使用
- 成本最低
- 事务数据最合适
- 和 MyBatis-Plus 最匹配

结论：

- **业务数据继续用 MySQL**

---

### 3.3 Prompt 模板管理

推荐：

- 主存：`MySQL`
- 缓存：`Redis`
- 兜底：`YAML`

原因：

- Prompt 模板是配置型数据
- 需要：
  - 场景区分
  - 版本管理
  - 启停控制
  - 审计字段
- MySQL 很适合这类管理

不建议：

- 只放 `yaml`
- 只放 `Redis`
- 为 Prompt 单独引入 MongoDB

结论：

- **Prompt 模板主存 MySQL，Redis 做缓存**

---

### 3.4 会话记忆与缓存

推荐：

- `Redis`

适合存：

- 聊天上下文
- Prompt 缓存
- 热点分析结果缓存
- 短期用户状态

结论：

- **Redis 继续保留，并作为 AI 会话与缓存层**

---

## 4. 知识库 / 向量库选型对比

这是你后续最需要认真权衡的一层。

### 方案 A：PostgreSQL + pgvector

适合：

- 知识库第一版
- 中小规模文档量
- 成本敏感
- 希望技术栈尽量简单

优点：

- 成本低
- 部署简单
- 生态成熟
- Spring AI 支持直接接
- 向量检索能力足够支撑第一版 RAG

缺点：

- 大规模高并发下不是最强方案
- 混合检索能力不如 Elasticsearch 灵活

推荐指数：

- **非常推荐，适合你当前阶段**

---

### 方案 B：Elasticsearch / OpenSearch

适合：

- 文档检索较复杂
- 既要关键词检索，又要语义检索
- 要做筛选、排序、搜索体验优化

优点：

- 关键词检索强
- 多字段过滤强
- 混合检索更灵活
- 文档搜索体验更成熟

缺点：

- 运维成本高于 pgvector
- 系统复杂度更高
- 作为第一版知识库有点偏重

推荐指数：

- **中期可考虑**

---

### 方案 C：Milvus / Qdrant

适合：

- 知识库量很大
- 向量检索是核心能力
- 希望后续扩展成 AI 检索平台

优点：

- 专门做向量检索
- 检索性能强
- 可扩展性更好

缺点：

- 引入成本更高
- 学习和运维成本更高
- 对你当前项目阶段偏重

推荐指数：

- **后期再考虑**

---

### 方案 D：Pinecone 等托管向量库

适合：

- 希望少运维
- 团队偏向云服务
- 预算允许

优点：

- 托管省事
- 上手快
- 扩容方便

缺点：

- 成本较高
- 数据和网络依赖外部平台
- 对国内环境和部署策略要谨慎

推荐指数：

- **可选，但不是当前首选**

---

## 5. 不推荐的方案

### 5.1 直接把知识库存 MySQL

问题：

- 可以存原始文档元数据
- 但不适合做真正的向量检索主方案

结论：

- **不推荐把 MySQL 当知识库向量检索主引擎**

### 5.2 Redis 直接当知识库主库

问题：

- 更适合缓存和临时数据
- 不适合做知识库主真相源

结论：

- **不推荐**

### 5.3 只靠 Prompt，不做 RAG

问题：

- 数据量一大就不稳定
- 上下文长度受限
- 结果容易漂移

结论：

- **知识库场景最终还是需要 RAG**

---

## 6. 针对你项目的推荐路线

### 阶段 1：当前最推荐

- 应用编排：`Spring AI`
- 业务主库：`MySQL`
- Prompt 模板：`MySQL + Redis 缓存 + YAML 兜底`
- 会话缓存：`Redis`
- 知识库：`PostgreSQL + pgvector`

结论：

- **这是最适合你当前项目阶段的组合**

---

### 阶段 2：文档检索需求变复杂

可以升级为：

- 保持 `Spring AI + MySQL + Redis`
- 知识库改为或新增：
  - `Elasticsearch / OpenSearch`

适用场景：

- 多字段文档检索
- 关键词 + 向量混合检索
- 检索结果体验要求更高

---

### 阶段 3：知识库规模显著增大

可以考虑：

- `Milvus / Qdrant`

适用场景：

- 文档量大
- 检索压力高
- 多知识库
- 多租户
- 平台化需求明显

---

## 7. 当前最建议你怎么选

如果你现在就要做下一阶段规划，我建议直接按下面这个版本确定：

### 核心推荐方案

- `Spring AI`
- `MySQL`
- `Redis`
- `PostgreSQL + pgvector`

### 这套方案的职责分工

- `MySQL`
  - 业务数据
  - Prompt 模板
  - AI 历史记录
  - AI 调用日志

- `Redis`
  - 聊天上下文
  - 热点缓存
  - Prompt 缓存

- `PostgreSQL + pgvector`
  - 文档切片
  - embedding
  - 向量召回

- `Spring AI`
  - 聊天
  - 工具调用
  - RAG
  - 结构化输出
